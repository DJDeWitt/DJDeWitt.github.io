A Lesson On Class

An introduction to object-oriented programming, classes, and instance variables and methods

October 19, 2014

Have you ever thought about why we call computers and calculators what we call them? It’s because they, quite literally, compute and calculate. In other words, they are both a THING that DOES. They’re essential to our lives because they’re able to do actions - these computations and calculations - that would be way too difficult or time-consuming for a person to do on their own.

Your computer and your code may seem like they’re on opposite ends of the hardware-software spectrum, but this conceptualization (“a THING that DOES”) is actually applicable across the board. You see, we make things (computers, calculators, and code) to help us do things, and often those things we want to do have a real-world analog. For example, we could calculate arithmetic using stones or an abacus, but a calculator with pre-defined equations is significantly easier and more time-efficient to use. A calculator is an object, and it’s only one of many objects we can mimic using code. This is probably a good time to mention that by “a THING that DOES”, we really mean “an OBJECT that DOES”. In other words, we’re talking about object-oriented programming.

To understand object-oriented programming, we first have to understand how the terminology we use lines up with our conceptualization. Remember how we said the “THING” is the “OBJECT”? Well, your calculator isn’t the only one in the world; there are plenty of calculators exactly like it that all have the same functionality, look, and behavior. In Ruby (and many other object-oriented programming languages), we call this a Class.

A calculator Class would essentially be the blueprint for all calculator Objects. The Class defines exactly what variables and methods an Object of that Class will have access to. These variables are essentially what the Object “remembers”, and, in the sense of a calculator, it would be good if our calculator could remember the numbers it is trying to add, subtract, multiply, or divide together!

So what are the methods? Well, methods are essentially verbs, they are the “DOES” of our statement. For a calculator, some handy verbs would be “adds”, “subtracts”, and “multiplies”. With these variables and methods, we can guarantee that all of our calculators will be able to perform basic arithmetic.

Now that we’ve explained it, let’s take a look at our calculator Class, which currently can only add a new number to an existing number. As you can see below, each new Object of this Class should initialize with a value, which we can then manipulate using the #add method. We can also assign a new value using the #set_value method, and get the current value using the #get_value method. 

      class Calculator
        def initialize(input_value)
          @value = input_value
        end

        def set_value(input_new_value)
          @value = input_new_value
        end

        def get_value
          return @value
        end

        def add(number_to_add)
          @value = @value + number_to_add
        end
      end

You'll notice that we have some variables with an "@" in front of them and some without this symbol. Variables with an "@" at the beginning are called instance variables, and they can be accessed from anywhere within the class, whereas variables without the "@" can only be accessed from within that specific method. This means that @value can be accessed everywhere, but input_new_value can only be accessed in the #set_value method!

You’ll notice that we have to have unique methods just for getting our current value and resetting it to a new value. You can imagine that if we allowed our Class to remember lots of values instead of just this one, we would quickly run into way too many lines of code, simply for getting access to and manipulating our variables. Thankfully, Ruby has a solution in the form of attr_ methods. The three attr_ methods look like this: 
      attr_reader :name
      attr_writer :best_friend
      attr_accessor :age

The attr_reader method creates a variable that you can output, or "read", but can't change. The attr_writer method creates a variable you can read, but you can change. The attr_accessor method creates a variable you can change and read. Now, let’s try rewriting our previous calculator Class using attr_ methods… 

      class Calculator
      attr_accessor :age

        def initialize(input_value)
          @value = input_value
        end

        def add(number_to_add)
          @value = @value + number_to_add
        end
      end

As you can see, it’s quite shorter! Obviously, this is a fairly simple class, but you can quickly see how important attr_ methods are when dealing with Classes of increasing complexity.
